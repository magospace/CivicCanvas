# Texas Public Data Explorer Skill

## Purpose

This skill helps Codex or another coding agent build Texas Data Canvas safely and consistently.

Texas Data Canvas turns Texas public datasets into interactive, source-cited dashboards using maps, charts, tables, filters, summaries, and safe query workflows.

## Use this skill when

Use this skill for tasks involving:

- Texas public datasets
- open data portals
- dataset discovery
- metadata inspection
- bounded queries
- MCP server tools
- canvas block generation
- public-data visualization
- source attribution
- data governance
- Miro export specs

## Core workflow

When implementing a feature:

1. Identify the user-facing workflow.
2. Identify the dataset/source involved.
3. Check or update the approved dataset catalog.
4. Validate the query shape with BoundedQuerySpec.
5. Render results through approved canvas blocks.
6. Include source attribution.
7. Include caveats.
8. Add or update tests.
9. Update docs if behavior changes.

## Approved visual blocks

Only these blocks may be generated by CanvasSpec:

- SummaryBlock
- MetricBlock
- ChartBlock
- MapBlock
- TableBlock
- FilterBlock
- SourceMethodBlock
- DatasetCardBlock

Do not implement arbitrary runtime execution of AI-generated HTML, JavaScript, SQL, or external scripts.

## Query safety rules

The app must:

- Use approved datasets only.
- Validate dataset IDs.
- Validate field names.
- Validate operators.
- Enforce row limits.
- Prefer aggregation.
- Hide or aggregate sensitive fields.
- Store query audit metadata.
- Return caveats with query results.

## Visualization rules

Use:

- Line chart for time series.
- Bar chart for ranked categories.
- Map/geography block for geographic data.
- Table block for inspectable results.
- Metric cards for high-level totals.
- SourceMethodBlock for attribution and methodology.

Always include SourceMethodBlock.

## Summary rules

Summaries should be factual, source-linked, and descriptive.

Avoid:

- unsupported causation
- policy recommendations presented as facts
- claims beyond the dataset
- hiding uncertainty
- implying sample data is live data

## Miro export rules

Miro export is optional/stretch.

When implementing Miro export:

1. Generate MiroExportSpec from an existing safe CanvasDocument.
2. Include a required source/method frame.
3. Do not write to Miro without user confirmation.
4. Do not export sensitive hidden fields.
5. Prefer aggregate visuals and tables.
6. Keep Texas Data Canvas as the data-query and governance layer.

## Example workflow: Dallas 311

User prompt:

> Show Dallas 311 service requests by category and ZIP code for 2024.

Expected implementation:

1. Select Dallas 311 dataset from catalog.
2. Inspect date, category, ZIP, status, and response-time fields.
3. Query grouped counts by category and ZIP.
4. Generate:
   - SummaryBlock
   - MetricBlock(s)
   - ChartBlock for requests over time
   - MapBlock for ZIP-level counts
   - ChartBlock for top categories
   - TableBlock for grouped results
   - SourceMethodBlock
5. Add filters for date range, category, status, and ZIP code.

## Example workflow: Austin permits

User prompt:

> Show Austin building permits by month and ZIP code.

Expected implementation:

1. Select Austin building permits dataset.
2. Query permit counts by month and ZIP.
3. Generate:
   - MetricBlock for total permits
   - ChartBlock for monthly trend
   - MapBlock for ZIP-level permits
   - TableBlock for top ZIP codes
   - SourceMethodBlock
4. Include caveat that permit records reflect administrative data and may not represent construction starts.
