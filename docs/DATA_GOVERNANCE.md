# Data Governance

## Source policy

Allowed:

- Public Texas city/state datasets
- Official open data portals
- Public APIs
- Static sample data derived from or representing public dataset structure with attribution

Disallowed:

- Scraping behind authentication
- Private datasets
- Sensitive personal data
- Unattributed data
- Arbitrary external URLs supplied by the model

## Field policy

Field classification:

- safe_public
- safe_with_aggregation
- sensitive_hide
- unknown_review

Default handling:

- Names: hide
- Emails/phones: hide
- Exact addresses: hide or generalize
- Latitude/longitude: aggregate/fuzz for sensitive topics
- ZIP code/city/county: allowed
- Request/permit IDs: allow only when clearly public and useful
- Public safety incident details: aggregate by geography/time/category
- Transportation incident details: aggregate by ZIP/time/type/status; hide precise street addresses unless a later governance review explicitly approves a generalized field.

## Query policy

The app must:

- Use approved datasets only.
- Validate dataset IDs.
- Validate field names.
- Validate operators.
- Enforce row limits.
- Prefer aggregation.
- Hide or aggregate sensitive fields.
- Store query audit metadata.
- Return caveats with query results.

Default limits:

- Max raw rows: 100
- Max aggregate rows: 1000
- Max sample rows: 25
- Max dashboard blocks generated by one prompt: 10

## Live adapter policy

Live adapters must generate source-specific query fragments from catalog-owned mappings only. The Socrata adapter currently allows:

- direct field identifiers matching `[A-Za-z_][A-Za-z0-9_]*`
- the catalog-owned month expression `date_trunc_ym(<identifier>)`

Other expressions such as `lower()`, `upper()`, `substr()`, arbitrary SoQL snippets, and user/model-provided expressions are rejected unless a future governance review adds explicit documentation and tests.

## Attribution policy

Every dashboard must show:

- Dataset name
- Source portal
- Dataset URL if available
- Date accessed
- Fields used
- Filters applied
- Query method
- Caveats
- License/terms when available

## Summary policy

Summaries must:

- Be descriptive
- Avoid unsupported causation
- Mention caveats
- Identify source dataset
- State if data is sample/demo data

## Sample provenance and persistence

Sample files under `data/samples/*.sample.json` are checked-in synthetic development samples aligned to approved public-data schemas. They are not complete live extracts or source-owned production snapshots. Use `docs/SAMPLE_AND_PERSISTENCE_REALNESS.md` for the current row-count, provenance, live/fallback, hidden-field, saved-canvas, share-link, seed API, and release-evidence matrix.

## Miro export policy

Miro export must:

- Include source/method/caveats frame.
- Not export sensitive hidden fields.
- Not create a board without user confirmation.
- Preserve dataset and query attribution.
- Prefer aggregate results over raw records.

## Houston pilot policy

Houston transportation is sample-first. The preferred source boundary is Houston TranStar traffic feed documentation and sample JSON, but no live adapter is promoted until stable live feed access and aggregate-safe field mappings are verified. The City Active Incidents page is not scraped into the app. The `precise_address` field is classified `sensitive_hide`; it must remain unavailable to bounded queries, dashboard blocks, CSV/JSON exports, Miro previews, gallery fixtures, and saved-canvas bundles.
